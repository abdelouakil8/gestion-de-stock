القسم 01

التشخيص — تقييم الوضع الحالي

بعد تحليل كامل للكود المصدري (الـ Backend، API Routes، الخدمات، النماذج، وشاشات PySide6)، الخلاصة العامة: بنية الـ Backend قوية وفوق المستوى المتوسط لمثل هذا النوع من المشاريع. المشكلة الحقيقية هي فجوة بين ثراء الـ Backend وضعف الواجهة الأمامية. عدة وظائف كاملة في جهة الخادم ليس لها واجهة مستخدم.

✓ ما يعمل بشكل جيد
  • بنية API-first صحيحة
  • Backend متكامل للمشتريات والمخزون
  • 13+ Endpoint إحصائي غني
  • نموذج StockMovement متين
  • Softdelete + Multi-tenant جاهز
  • Decimal لجميع المبالغ المالية
  • التحقق من المخزون في جهة الخادم
⚠ الفجوات الحرجة
  • المشتريات: Backend كامل، لا شاشة
  • حركة المخزون: خدمة جاهزة، واجهة غائبة
  • الإحصائيات: 4 من 13 Endpoint فقط مستخدمة
  • الموردين: لا يظهر الرصيد المستحق
  • تعليق الفاتورة (Park Sale): غير موجود
  • التنبيهات: لا يوجد تصنيف حسب الخطورة
✗ أخطاء يجب تجنبها الآن
  • إضافة مستخدمين متعددين حالياً
  • Backup سحابي (شبكة غير موثوقة)
  • SMS تلقائي (تكلفة + تعقيد)
  • تواريخ صلاحية (ميزة تخصصية)
  • نقاط ولاء (غير شائعة في الجزائر)
  • ربط ميزان/كاشير (تخصصي جداً)
رقم رئيسي: نصف الوظائف المفقودة في الـ UI موجودة بالفعل في الـ Backend. الأولويتان المطلقتان (المشتريات وسجل الحركة) لا تحتاجان أي نموذج أو خدمة إضافية — فقط عمل على الواجهة الأمامية.

القسم 03

تحليل مفصل — شاشة بشاشة

🛒

الكاش (Caisse)

حافظ · طور

الشاشة الأكثر استخداماً — 100% من المعاملات تمر عبرها. أي احتكاك هنا يكلف وقتاً حقيقياً.

  • ✅الحفاظ على: بحث المنتج (باركود + اسم)، مستويات الأسعار، ربط العميل، F12 للدفع، إيصال PDF.
  • 🔧تحسين: إظهار رصيد العميل المتبقي باللون الأحمر قبل التأكيد.حالياً يصعب ملاحظته. التاجر يحتاج معرفة فورية إذا كان العميل مديوناً.الأولوية 1
  • ➕إضافة — تعليق الفاتورة (Park Sale): تعليق الفاتورة الحالية وبدء أخرى. استرجاع المعلقة بنقرة.مقاطعات مستمرة في التجارة الجزائرية. بدونها، الكاشير يفرغ السلة أو يجمد الكاش.الأولوية 1
  • ➕إضافة: إدخال الكمية قبل الإضافة (كتابة "5" ثم مسح = 5 وحدات).الأولوية 2
  • ➕إضافة: ربط مستوى السعر تلقائياً بنوع العميل.الأولوية 2
  • ➕إضافة: إعادة طباعة آخر إيصال (زر أو اختصار).بسيط
  • ❌لا تضف الآن: شبكة منتجات بأيقونات لمسية — مناسبة للتابلت، تثقل واجهة PC.
📦

المخزون (Stock)

طور · أكمل
  • ✅الحفاظ على: قائمة منتجات، بحث، صور، أسعار بمستويات، بطاقة تفاصيل مع إحصائيات، أرشفة.
  • ➕إضافة — علامة تبويب "سجل الحركة": عرض حركة المخزون للمنتج (بيع، شراء، إرجاع، تعديل).نموذج StockMovement وخدمة stock_movements.py كاملان. مجرد عمل UI.Backend جاهز
  • ➕إضافة: خط حد التنبيه بلون — برتقالي/أحمر على الصفوف تحت الحد.الأولوية 2
  • ➕إضافة: مرشحات سريعة حسب التصنيف (أزرار chips).
  • ➕إضافة: تعديل مخزون سريع من القائمة (أيقونة → نافذة "تصحيح مخزون").
💡 إجراء فوري: علامة التبويب "سجل الحركة" لا تحتاج أي Backend جديد — stock_movements.py جاهز.
🚚

المشتريات (أوامر الاستلام) ★ الأولوية المطلقة

شاشة مفقودة

هذه هي الثغرة رقم 1 في المشروع. التاجر يشتري بضاعة عدة مرات في الأسبوع. لا توجد أي شاشة لإدارة هذا رغم أن الـ Backend كامل 100%.

✅ الـ Backend كامل: purchase_orders.py, purchasing.py, نماذج PurchaseOrder + PurchaseOrderItem + SupplierPayment. API تقدم: list, get, create (مع تحديث آني للمخزون), add_payment. العمل المتبقي: فقط الـ UI.

ما يجب أن تحتويه الشاشة:

  • 1قائمة أوامر الاستلام — التاريخ، المورد، المبلغ، المدفوع، الباقي.
  • 2إنشاء أمر: اختيار مورد ← إضافة منتجات ← تأكيد ← تحديث المخزون.
  • 3تسجيل دفعة مورد — دفع جزئي أو كلي. إظهار الرصيد المتبقي.
  • 4مرشحات: بالمورد، بالحالة (مدفوع/جزئي/غير مدفوع)، بالفترة.
  • 5إجمالي المبالغ المستحقة — رقم معروض في رأس القائمة.
خيار معماري: دمج المشتريات في شاشة الموردين (علامة تبويب "أوامر") بدلاً من إنشاء شاشة تاسعة.
👥

العملاء (Clients)

طور
  • ✅الحفاظ على: اللوحة اليمنى (إحصائيات + تاريخ مشتريات + فواتير + دفعات)، ترتيب أفضل العملاء.
  • ➕إضافة — مستوى السعر الافتراضي: معامل على البطاقة ("يشتري بـ: ديتاي / جروج / سوبر جروج"). يُفعَّل تلقائياً في الكاش.تأثير كبير. تاجر الجملة لا يجب أن يغير السعر يدوياً كل مرة.الأولوية 1
  • ➕إضافة: مؤشر "آخر شراء منذ X يوم" على كل سطر.
  • ➕إضافة: مرشح "عملاء غير نشطين منذ 30 يوماً".
🧾

المبيعات (Ventes)

طور
  • ✅الحفاظ على: قائمة مبيعات + مرشحات، تفاصيل فاتورة، إعادة طباعة، ربط عميل.
  • ➕إضافة — شريط ملخص في الرأس: "اليوم: X فاتورة · الإجمالي: XX XXX دج · غير المدفوع: X XXX دج".رقم أساسي يبحث عنه التاجر أولاً. يجب أن يكون مرئياً بدون تمرير.الأولوية 1
  • ➕إضافة: إرجاع سريع من القائمة (زر على الفاتورة → فتح نافذة الاسترجاع).
  • ➕إضافة: تصدير Excel للقائمة المفلترة (Endpoint موجود).
🏭

الموردين (Fournisseurs)

دمج مع المشتريات
  • ✅الحفاظ على: قائمة موردين، بطاقة تفاصيل، تاريخ طلبات.
  • 🔧تحسين: إظهار بأرقام كبيرة: إجمالي المشتريات، المدفوع، الرصيد المتبقي.خدمة purchasing.py تحسبها. مجرد عرض مفقود.الأولوية 1
  • ➕إضافة — علامة تبويب "أوامر الاستلام": قائمة أوامر + زر "أمر جديد" + تسجيل دفعة.يتجنب شاشة تاسعة. المشتريات سياقية لمورد معين.
التوصية: إعادة تسمية "الموردين" → "المشتريات والموردين". شاشة بعلامتي تبويب: "الموردين" و"أوامر الاستلام".

القسم 07

خطة التطوير — MVP vs المرحلة الثانية vs التأجيل

معايير الأولوية: التأثير اليومي × تكلفة التطوير. كل ما له Backend جاهز هو أولوية مطلقة.

✅ MVP — يجب فعله الآن (حسب الأولوية)
  • Backend جاهز شاشة المشتريات ضمن الموردين
  • Backend جاهز علامة تبويب حركة المخزون
  • Backend جاهز لوحة "المخزون الراكد" في التنبيهات
  • Backend جاهز LineChart تطور يومي في الإحصائيات
  • Backend جاهز أفضل العملاء في الإحصائيات
  • Backend جاهز إظهار رصيد المورد
  • UI فقط تعليق الفاتورة (Park Sale)
  • UI فقط شريط ملخص في المبيعات
  • UI فقط ألوان خطورة في التنبيهات
  • Backend صغير مستوى السعر الافتراضي للعميل
  • UI فقط رصيد العميل في الكاش
  • UI فقط أقسام قابلة للطي في الإحصائيات
🔵 المرحلة الثانية — بعد الاستقرار
  • Backend جديد المصروفات (نموذج + خدمة + API + UI)

أفضل 10 — ما يصنع منتجاً احترافياً

هذه التحسينات ليست وظائف تجارية — هي الفرق بين "إنه يعمل" و"إنه احترافي". كل منها يمكن تسليمه في أقل من يوم.

  1. إشعارات (Toast) متناسقة لجميع الإجراءات كل إنشاء/تعديل/حذف → إشعار واضح. أبداً لا صمت بعد إجراء. الإشعارات هي إشارات الثقة في الـ UI.
    تأثير فوري
  2. حالات فارغة مصممة بعناية (Empty States) لا قائمة فارغة بدون رسم + رسالة + زر إجراء. "لا موردون" → أيقونة + "أضف أول مورد" + زر.
    حاسم لـ UX
  3. اختصارات لوحة المفاتيح معروضة وموثقة F1-F8 للتنقل، F12 للدفع، Échap للإغلاق. عرضها في الواجهة يجعل التطبيق احترافياً.
  4. مؤشرات تحميل لجميع طلبات API كل استدعاء شبكة → spinner أو skeleton screen. أبداً شاشة تتجمد بصمت.
    موجود جزئياً
  5. تأكيد ذكي قبل الحذف "حذف المنتج حليب كانديا 1L؟" أبداً "حذف؟". التأكيد حماية وليس عائقاً.
  6. العملة بالدينار الجزائري (دج) في كل مكان "12 500 دج" وليس "12500". رقم بدون عملة هو مصدر خطأ.
    غير قابل للتفاوض
  7. بحث عام سريع (Ctrl+K) من أي شاشة: اكتب منتجاً أو عميلاً → اذهب مباشرة.
    المرحلة الثانية
  8. وضع داكن وثيق، ليس مجرد "خلفية سوداء" تحقق من كل نافذة وكل جدول — لا ألوان قاسية (#000/#fff) على الأسطح.
    متقدم بالفعل
  9. أرقام فواتير متصلة وقابلة للقراءة "FAC-2026-0042". مطبوع على الإيصال + مرئي في القائمة. التاجر يسترجع الفاتورة برقمها.
  10. مقاومة أخطاء الشبكة / API البطيء كل شاشة تظهر حالة التحميل، تتعامل مع timeout، وتقترح "إعادة المحاولة".
    متانة

المشروع في وضع يحسد عليه: بنية تقنية قوية، API-first صحيحة، نماذج مصممة جيداً. الأولوية ليست إضافة وظائف — بل ربط ما هو موجود بالفعل. أول 7 مهام في الـ MVP لا تلمس الـ Backend.

المخاطر الرئيسية: الانبهار بالوظائف المتقدمة (مستخدمين متعددين، Backup سحابي، SMS) قبل أن يكون قلب المنتج لا يُعلى عليه. التاجر الذي لا يجد رصيد مورديه في ثانيتين لن يعود، حتى لو كان المنتج يقدم نقاط ولاء.

ابنِ أولاً ما يلمسه التاجر كل صباح: الكاش، المشتريات، ملخص اليوم، التنبيهات. اجعله سريعاً وموثوقاً ومقروءاً. الباقي يأتي بعد ذلك.
